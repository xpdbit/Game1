# Game1 长期演进路线图

**项目**: Game1 - Unity 6 挂机放置类游戏
**创建时间**: 2026-04-26
**基于**: 项目现状分析 + 2025年Unity开发最佳实践

---

## 一、项目现状总结

### 1.1 技术栈
| 组件 | 版本 |
|------|------|
| Unity | 6000.1.5f1 |
| URP | 17.1.0 |
| C# | 9.0 |
| DI框架 | VContainer |
| 输入系统 | Input System 1.14.0 |
| 透明窗口 | UniWindowController 0.9.8 |

### 1.2 代码架构（68个脚本）
```
Assets/Scripts/
├── Core/           # 核心系统（GameLoop/SaveSystem/EventBus/Input/Debug/Utils）
├── Entities/       # 实体（Player/World/NPC/Actor）
├── Modules/        # 游戏模块（Idle/Travel/Combat/Team/Inventory/Skill/Card/Prestige/PVP/Pet/Activity）
├── Events/         # 事件系统（EventQueue/EventChain/EventManager/EventTreeManager/EventTreeRunner）
├── Roguelike/      # 随机地图生成
└── UI/             # UI系统（UIManager/面板组件/布局/列表）
```

### 1.3 已有设计模式
- **Singleton**: 各Manager使用instance模式
- **IModule接口**: PlayerActor通过ModuleCollection管理模块
- **事件驱动**: EventBus发布-订阅模式
- **状态机**: TravelState/UIState等枚举状态

## 二、长期演进路线图（6大阶段 × 3子任务）

---

### 阶段一：核心玩法深化

#### 1.1 战斗系统重构
**目标**: 将CombatSystem完全融入模块系统，支持多目标战斗和技能系统

**子任务**:
1. **战斗系统模块化** - 将CombatSystem重构为IModule实现，支持Tick/Dynamic注册
2. **多目标战斗** - 扩展ExecuteMultiEnemyCombat，支持队伍系统联动
3. **技能与战斗联动** - SkillDesign触发CombatSystem，支持Passive/Active/Ultimate技能

**验收标准**:
- [ ] CombatSystem实现IModule接口
- [ ] 队伍成员参与多目标战斗
- [ ] 技能可以触发战斗事件
- [ ] 战斗日志完整记录伤害/暴击/闪避

---

#### 1.2 事件系统增强
**目标**: 完善事件树（EventTree）和分支叙事，支持玩家选择影响故事

**子任务**:
1. **EventTreeRunner完善** - 支持分支选择、历史返回、随机开始
2. **事件树配置编辑器** - 创建Unity编辑器扩展，可视化编辑EventTree
3. **选择结果持久化** - 玩家选择记录到存档，影响后续事件

**验收标准**:
- [ ] EventTree支持Root/Choice/Random/End节点类型
- [ ] 玩家可以回退到历史选择点
- [ ] EventTreeRunner支持从随机节点开始
- [ ] 选择结果写入PlayerSaveData

---

#### 1.3 轮回与卡牌系统
**目标**: 完成Prestige轮回系统和Card抽卡系统

**子任务**:
1. **PrestigeManager实现** - 点数商店/资源保留/轮回升级
2. **CardDesign + CardManager** - 卡牌核心逻辑和静态API
3. **卡牌与战斗联动** - 卡牌提供战斗加成/特殊技能

**验收标准**:
- [ ] PrestigeManager支持多轮回层级
- [ ] 卡牌系统支持N/R/SR/SSR/UR/GR稀有度
- [ ] GachaType支持单抽/十连
- [ ] 卡牌效果生效于战斗/旅行/收益

---

### 阶段二：系统解耦与重构

#### 2.1 VContainer DI重构
**目标**: 全面使用VContainer替代手动Singleton，提升代码可测试性

**子任务**:
1. **移除硬编码Singleton** - 各Manager通过构造函数注入
2. **Service Locator模式** - GameMain作为ServiceLocator统一管理服务
3. **Lifetime Scope设计** - 按场景/全局划分依赖生命周期

**验收标准**:
- [ ] 所有Manager实现构造函数注入
- [ ] GameMain通过Container.Resolve获取服务
- [ ] 支持MonoBehaviour的LifetimeScope嵌套
- [ ] 单元测试可Mock Manager依赖

---

#### 2.2 统一事件系统
**目标**: 整合EventBus/InventoryEventBus/TeamEventData为统一事件框架

**子任务**:
1. **EventBus扩展** - 支持泛型事件类型和强类型事件参数
2. **事件追溯系统** - 事件链记录（EventChain）支持Undo/Redo
3. **模块间通信标准化** - 所有模块通过事件通信，消除直接引用

**验收标准**:
- [ ] EventBus支持`EventBus<T>.Publish(T event)`强类型
- [ ] 事件系统记录完整事件链
- [ ] 支持按时间顺序回溯事件
- [ ] 模块间零直接引用，全通过事件通信

---

#### 2.3 模块架构标准化
**目标**: 所有游戏模块遵循`Design + Manager + Data`模式

**子任务**:
1. **模块模式推广** - Team/Inventory/Card/Skill等模块统一设计
2. **模块生命周期管理** - IModule接口标准化（Initialize/Tick/Activate/Deactivate）
3. **模块热插拔** - 支持运行时启用/禁用模块

**验收标准**:
- [ ] 所有模块实现IModule接口
- [ ] 模块支持动态注册/注销
- [ ] PlayerActor.modules提供GetModule<T>()查询
- [ ] 模块配置文件支持热更新

---

### 阶段三：性能优化

#### 3.1 UI渲染优化
**目标**: 降低Draw Call，提升UI渲染效率

**子任务**:
1. **Canvas动静分离** - 静态UI和动态UI分属不同Canvas
2. **纹理图集制作** - UI元素打包到共享图集
3. **TextMeshPro优化** - 数字跳动避免GC，使用对象池

**验收标准**:
- [ ] 战斗场景UI Draw Call < 50
- [ ] UI渲染时间 < 2ms
- [ ] Material切换 < 30次/帧
- [ ] 文本更新无GCAlloc

---

#### 3.2 内存管理与资源加载
**目标**: 优化内存占用，减少卡顿

**子任务**:
1. **Addressables迁移** - 配置资源使用Addressable Asset System
2. **资源生命周期管理** - 异步加载+对象池复用
3. **内存泄漏检测** - 实现资源释放追踪

**验收标准**:
- [ ] 大资源异步加载无主线程阻塞
- [ ] 场景切换内存峰值 < 200MB
- [ ] 对象池覆盖80%动态对象
- [ ] 无未释放的Texture/Material引用

---

#### 3.3 存档系统完善
**目标**: 实现完整的存档/读档功能，支持离线数据

**子任务**:
1. **存档数据结构重构** - GameSaveData包含完整游戏状态
2. **增量存档** - 仅保存变更数据，减少IO
3. **云存档支持** - 预留接口支持云端同步

**验收标准**:
- [ ] 存档包含Player/World/Inventory/Team/Skill等完整状态
- [ ] 离线时间计算准确，收益正确
- [ ] 存档文件 < 1MB（压缩后）
- [ ] 读档时间 < 2秒

## 五、风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 透明窗口性能问题 | UI渲染延迟 | Canvas动静分离，禁用全屏半透明 |

| CI构建失败 | 发布阻塞 | 构建失败立即通知，保持主分支绿色 |

## 六、设计迭代（第1轮回审查）

### 6.1 实现经验总结

#### 架构评估
1. **IModule接口** - 设计良好，已被CombatModule、IdleRewardModule等采用
2. **Design+Manager模式** - Team/Inventory/Card遵循良好，Prestige命名差异可接受
3. **VContainer混合模式** - 手动创建+VContainer解析，适合Unity游戏

#### 需要改进的设计
1. **存档系统** - 需要支持更多模块的序列化（Skill、Card、Team等）
2. **EventTree配置** - 缺少编辑器工具，可视化编辑困难
3. **卡牌系统** - 卡牌效果需要更灵活的效果系统

### 6.2 设计改进建议

#### 建议A：增强效果系统
**问题**：当前卡牌/事件使用硬编码的Effect类型
**方案**：实现可扩展的效果系统，支持组合效果

#### 建议B：事件可视化编辑器
**问题**：EventTree XML配置难以维护
**方案**：开发Unity编辑器扩展，支持图形化编辑

#### 建议C：模块配置ScriptableObject
**问题**：当前模块配置分散在代码中
**方案**：使用ScriptableObject集中管理模块配置

### 6.3 下轮回优先级调整

基于实现经验，调整优先级：
1. **P0**: UI渲染优化（玩家直接感知）
2. **P1**: 事件树编辑器（提升开发效率）
3. **P1**: 效果系统增强（支持更复杂设计）
4. **P2**: 自动化测试（保障代码质量）
