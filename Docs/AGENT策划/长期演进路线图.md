### 1.2 架构问题清单

**严重问题 (P0):**
1. PlayerActor双实例 - GameMain和GameLoopManager各创建一个
2. Save()/Load()未实现 - 存档功能不可用
3. GameLoopManager未真正接入游戏循环
4. GameMain.Update()未调用GameLoopManager.Tick()

**中等问题 (P1):**
1. GameDebug未接入游戏循环
2. 离线收益未正确应用护肝机制
3. 金币float→int精度丢失
4. 自动存档首次运行不触发

### 1.3 架构评估

**优点**:
- 模块化设计，核心逻辑与MonoBehaviour分离 ✓
- 良好的设计模式应用（单例、观察者、模块集合） ✓
- InventoryDesign/TeamDesign模式可复用 ✓
- 事件驱动通信 ✓

**缺点**:
- 过度使用单例，耦合度高 ✗
- 缺乏依赖注入框架 ✗
- 无自动化测试 ✗
- 无CI/CD ✗
- UI缺乏数据绑定/MVVM ✗
- 无对象池 ✗
- 无性能分析 ✗
- Save系统不完整 ✗

## 二、研究成果与最佳实践

### 2.1 参考项目分析

| 项目 | 架构特点 | 可借鉴点 |
|------|----------|----------|
| LumberjackIdle (Unity 6) | VContainer DI + 状态机 + 对象池 | DI架构、对象池 |
| Cookie Clicker Clone | Clean Architecture + MVC + TDD | 架构分层、测试 |
| Idle-Miner | Zenject + UniRx + 数据驱动 | 响应式编程、配置分离 |
| Unity-Modular-Game-Architecture | VContainer + FSM + EventBus | 模块化架构 |
| EasyCS | ECS hybrid + 数据驱动 | ECS过渡策略 |

### 2.2 推荐技术栈

| 领域 | 当前 | 建议 | 理由 |
|------|------|------|------|
| DI框架 | 无 | VContainer | 轻量、自动化、文档好 |
| 动画 | 手动 | PrimeTween/DOTween | 性能好、易用 |
| UI模式 | 直接控制 | MVVM + DataBinding | 分离逻辑视图 |
| 状态机 | Switch | FSM框架 | 可追踪、可测试 |
| 对象池 | 无 | XObjectPool | 减少GC |
| 序列化 | JSON | Json.NET + ScriptableObject | 功能更强 |
| 日志 | Debug.Log | Unity.Logging/NLog | 分级、可配置 |
| CI | 无 | GitHub Actions | 免费、集成好 |
| 测试 | 无 | Unity Test Framework | 官方支持 |

### 2.3 Unity 6 性能优化要点

1. **Draw Call优化**: URP Batcher > Static Batching > Dynamic Batching > GPU Instancing
2. **内存优化**: 对象池、结构体复用、避免装箱
3. **启动优化**: 场景异步加载、资源压缩、代码剥离分析
4. **ECS/DOTS**: 用于大量实体处理，保留MonoBehaviour用于UI
